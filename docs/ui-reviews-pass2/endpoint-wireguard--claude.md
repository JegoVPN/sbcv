Upstream authority: `docs/upstream/sing-box/testing/configuration/endpoint/wireguard.md` (since 1.11.0). ## Verdict (2-3 sentences)
The node renders and round-trips, and pass-1's two headline P1s are now STALE/FIXED: `peers` is a structured repeater (not a JSON textarea) and both `private_key` and `peers[].pre_shared_key` are masked via `SensitiveTextField`. But two real P0s remain: (1) the scaffold, the peers editor, AND the test all write `peers[].server` / `server_port`, which are the *outbound* schema — the endpoint requires `address` / `port`, so every WireGuard endpoint this app produces emits peer keys sing-box ignores; and (2) the "outbound behavior" half of the hybrid is missing — a WireGuard endpoint tag is a valid route target / selector & urltest member / dns detour, but it has zero input ports and is absent from the `outbound` reference paths, so it can never be wired as a downstream target on canvas and is dropped by rename/delete cascades. Required-field diagnostics (`address`, `private_key`, `peers`, `peers[].public_key`, `peers[].allowed_ips`) are still entirely absent.

## 3. Upstream/Downstream Links
Endpoints share the outbound tag namespace ("inbound and outbound behavior"), so a WireGuard endpoint should be referenceable everywhere an outbound tag is, plus own its detour.

- Endpoint's OWN detour → outbound: present. `referenceRegistry.ts:334` includes `/endpoints/*/detour` under the `outbound` kind; `replaceOutboundRefs`/`removeOutboundRefs` scan `config.endpoints[].detour` (`referenceRegistry.ts:166,187`); canvas edge built at `graph.ts:647-648`; diagnostic `missing-endpoint-detour` (`diagnostics.ts:111-120`). Correct.
- Endpoint referenced AS a route/proxy target — **all MISSING:**
  - `route.final` / `route.rules[].outbound`: the `outbound` reference kind resolves these to `config.outbounds` only; an endpoint tag placed in `route.final` is neither validated nor cascaded on rename/delete (`replaceOutboundRefs` `referenceRegistry.ts:157-176` never matches the endpoint as the *referent*, only fixes endpoints' own detour). MISSING link + cascade gap.
  - selector / urltest `outbounds[]`, `default`: same — endpoint tags are legal members but not modeled. MISSING.
  - `dns.servers[].detour`: same. MISSING.
- The `endpoint` reference kind (`referenceRegistry.ts:344-348`, paths `/dns/servers/*/endpoint`, `/services/*/verify_client_endpoint`, `/certificate_providers/*/endpoint`) is **Tailscale-only** semantics. A WireGuard endpoint is NOT referenced by any of those three (per upstream they are MagicDNS / DERP / cert-provider features). So for wireguard these paths are inert — harmless but means the only "endpoint" reference modeling in the app does not apply to wireguard at all.

Net: the dial-detour (outbound-side, endpoint→outbound) link is correct; every "endpoint-as-target" link (the outbound-behavior half) is missing. No spurious wireguard-specific links beyond the always-on Tailscale reference card (see P2-2).

## Findings (prioritized)
- **[P0-1] Peers write the outbound schema (`server`/`server_port`) instead of the endpoint schema (`address`/`port`).** Scaffold `commands.ts:571-572` seeds `server`/`server_port`; peers editor reads/writes `peer.server` (`Inspector.tsx:4616-4620`) and `peer.server_port` (`Inspector.tsx:4622-4628`); add-peer seeds them (`Inspector.tsx:4601-4602`); and the test *locks the bug in* by asserting `{ server: "192.0.2.1" }` (`tests/app.test.tsx:1403`). Per `docs/upstream/.../endpoint/wireguard.md:81-87` the endpoint peer keys are `address` and `port`; `server`/`server_port` are the legacy *outbound* wireguard keys (`outbound/wireguard.md:34-35`). Result: exported endpoints carry peer keys sing-box silently ignores (no peer address/port), so the tunnel is misconfigured. Fix: rename to `address`/`port` in scaffold, editor labels/handlers, and update the test assertion + seed.
- **[P0-2] WireGuard endpoint is not referenceable as a route/proxy target (missing the "outbound behavior" half).** No input ports (`tests/sbc-node-ports.test.ts:20`; relations `route-final`/`route-rule`/`selector`/`urltest`/`dns-server-detour` target outbounds only, `portRelationRegistry.ts:93,95,103,104,105`), and the `outbound` reference kind (`referenceRegistry.ts:333-337`) resolves `route.final`/`rules[].outbound`/selector/urltest/dns-detour against `config.outbounds` only, never `config.endpoints` (`replaceOutboundRefs` `referenceRegistry.ts:157-176`). So an endpoint tag used as a route target/selector member can't be wired on canvas and is silently dropped/un-renamed on cascade. Fix: add endpoint as a valid source for the outbound-target relations (input ports) and include `config.endpoints` tags in the `outbound` reference resolution.
- **[P1-1] No semantic diagnostics for required WireGuard fields.** `diagnostics.ts:111-120` only checks `detour`. Missing hard errors: empty `address`, empty/placeholder `private_key`, empty `peers`, any peer missing `public_key`, any peer missing `allowed_ips`. A blank/invalid WireGuard endpoint currently shows green. (pass-1 P2-1 still valid; should be P1 given P0-1 already produces broken configs.)
- **[P1-2] `peers[].persistent_keepalive_interval` is a free-text field that writes an invalid string.** `Inspector.tsx:4648-4659` placeholder `25s`; upstream is a number of seconds. Use a `type="number"` input coercing to `Number`.
- **[P1-3] Interface fields `system`, `name`, `mtu`, `listen_port`, `udp_timeout`, `workers` are not first-class.** They fall to the Advanced scalar fallback (`Inspector.tsx:4730`) as freeform text (so `mtu`/`listen_port`/`workers` are text, not number). Promote to a typed "Interface" sub-section (toggle for `system`, number inputs for `mtu`/`listen_port`/`workers`, text for `name`/`udp_timeout`). (pass-1 P2-5, broadened.)
- **[P2-1] No required (`*`) markers on `address`, `private_key`, `peers[].public_key`, `peers[].allowed_ips`.** Inspector treats required and optional fields identically (`Inspector.tsx:4575-4647`).
- **[P2-2] Tailscale-only "Connections" card renders for wireguard endpoints.** `Inspector.tsx:4554-4571` is gated only on `ref.kind === "endpoint"`, so a wireguard endpoint shows three always-"none" Tailscale rows. Gate on `entityType === "tailscale"`. (pass-1 P2-2 still valid.)
- **[P2-3] `peers[].reserved` has no control.** Not rendered in the peer row (`Inspector.tsx:4613-4669`); only survives if present in imported JSON. Add an optional 3-byte tuple input.
- **[P2-4] `EndpointConfig` type incomplete.** `types.ts:49-53` lacks `peers`, `system`, `name`, `mtu`, `listen_port`, `udp_timeout`, `workers`; add these + a `WireGuardPeer` type (`address`,`port`,`public_key`,`pre_shared_key`,`allowed_ips`,`persistent_keepalive_interval`,`reserved`). (pass-1 P2-3 still valid.)

Where pass-1 is now STALE: pass-1 P1-1 (peers raw `<textarea>`), P1-2 (`private_key` unmasked), and P2-4 (`pre_shared_key` unmasked) are all FIXED — a structured masked repeater shipped at `Inspector.tsx:4588-4675`. pass-1 did NOT catch the `server`/`server_port` key bug (P0-1) or the endpoint-as-route-target gap (P0-2); pass-1's "left port correctly empty" conclusion is WRONG given the hybrid outbound behavior.

SUMMARY: 2 P0, 3 P1, 4 P2.
