# External Config E2E Hardening

## Outcome

Harden SBC against real-world sing-box configs collected from GitHub. The goal is complete only after at least 200 unique external sing-box JSON configs/templates are accepted into the fixture manifest and every accepted fixture passes its assigned validation, import, graph-derivation, render, JSON round-trip, and export gate.

This goal is not about changing SBC into a traditional form UI. The product remains a Higgsfield-style React Flow visual editor backed by canonical JSON/domain state.

## Scope

In scope:

- GitHub-driven discovery of real sing-box JSON configs/templates.
- At least 200 unique accepted configs/templates from multiple repositories.
- Version-aware validation for sing-box 1.12, stable 1.13, and testing 1.14 where fixtures declare or imply a supported validation version. 1.11 fixtures remain UI/display coverage only.
- Stable-first release behavior remains unchanged: default templates and export must still target current stable.
- Fixture manifest with source repository, source path, source commit SHA, detected version/channel, transformations, expected gates, and final result.
- Deterministic local fixtures checked into the repo after discovery. CI must not depend on live GitHub during normal test runs.
- Mock server node injection for subscription templates that use placeholders such as `{all}` or provider-generated node pools.
- Data-driven unit/integration tests for import, normalization, graph derivation, diagnostics, JSON preservation, and export.
- Data-driven Playwright E2E tests for representative external fixtures.
- Compatibility report that separates official pass, legacy pass, display-only pass, and rejected/quarantined configs.
- Documentation of failures fixed during the goal.

Out of scope:

- Copying another project's UI layout.
- Treating external template extensions as official sing-box fields.
- Counting broken, duplicate, quarantined, non-JSON, or unreviewed configs toward the 200 accepted passing target.
- Runtime network fetch from random GitHub repos in the shipped frontend.
- Storing secrets, real credentials, private nodes, or user-specific endpoints.

Current implemented local gates:

- `pnpm fixtures:update:external` regenerates the deterministic external fixture corpus from public GitHub sources.
- `pnpm fixtures:update:external` also refreshes official gate metadata by running version-matched sing-box binaries when installed.
- `pnpm validate:external` runs manifest integrity checks, official `sing-box check` for every fixture with `official_check`, the external fixture unit gate, and a sharded JSDOM render gate for every accepted fixture.
- `pnpm test` includes all accepted external fixtures and currently checks JSON parse, canonical import, semantic diagnostics, graph derivation, JSON stringify/parse round-trip, export content, and the 200+ fixture count.
- `node scripts/run-external-render-gate.mjs` renders every accepted fixture through the App surfaces in bounded shards, checking the canvas, Inspector, JSON editor, and route/DNS table panels without holding all React Flow renders in one worker.
- `pnpm e2e` imports 20 representative external fixtures, asserts node classes, opens ordered rule tables, verifies the Higgsfield-style selected node DOM, verifies the top-right Inspector DOM, opens JSON Preview, exports JSON, re-imports the export, and fails on runtime page/console errors.
- `pnpm e2e` also runs a mobile viewport smoke for the selected node and floating Inspector.
- External fixture E2E currently writes a review screenshot at `test-results/sbc-selected-node-inspector.png` during local runs.
- Official external fixture checks target `sing-box-1.12`, `sing-box-stable`, and `sing-box-testing`. 1.11 fixtures are not official-check gates.
- 66 versioned fixtures are explicitly marked display/template-compatible because their matching official binary rejects them, with the failure reason stored in `fixtures/external/manifest.json`.
- Browser visual review has been performed against the local Vite app for the selected-node plus Inspector state.
- Large ordered `route.rules` / `dns.rules` lists remain table-owned. The canvas derives individual rule nodes only for compact rule lists, so real configs with hundreds of ordered rules still render the main Route/DNS relationship without flooding React Flow.
- Dense selector/urltest candidate lists remain tag-reference data in canonical JSON and Inspector fields. The canvas caps candidate edge rendering for large groups to preserve React Flow responsiveness while keeping export/import semantics complete.

## Fixture Acceptance Rules

An accepted fixture must:

- be valid JSON after deterministic placeholder replacement;
- be identifiable as a sing-box config or config template;
- have no real secrets committed;
- preserve unknown fields on import/export;
- generate at least one graph node or an explicit display diagnostic;
- not crash domain parsing, graph derivation, Inspector, rule tables, JSON preview, or export;
- pass every expected gate defined in the manifest.

Duplicate configs do not count. Duplicate detection uses normalized JSON hash after secret redaction and deterministic placeholder replacement.

Quarantined fixtures are allowed for investigation, but they do not count toward the 200 accepted passing target.

## Version And Gate Matrix

| Fixture class | Official binary gate | UI/import gate | Counts toward 200 |
| --- | --- | --- | --- |
| current stable config | matching `sing-box-stable check` | required | yes |
| testing / 1.14 config | matching `sing-box-testing check` | required | yes |
| explicit 1.13 config | sing-box 1.13.x check | required | yes |
| explicit 1.12 config | sing-box 1.12.x check | required | yes |
| explicit 1.11 config | UI/display/import compatibility only | not required | no |
| subscription template extension | official check if transformable; otherwise expected extension diagnostics | required | yes only if expected diagnostics pass |
| legacy config incompatible with current stable | version-matched old binary check or migration check | required | yes only if version gate passes |
| parse-fail / unsafe / duplicate | none | none | no |

The 200 target means 200 accepted fixtures pass their own manifest gates. It does not mean silently downgrading official validation to display-only.

## Known Fix Areas To Expect

From initial inspection, the implementation should be prepared to fix:

- `rule_set` as both `string` and `string[]`;
- `domain` as well as `domain_suffix` / `domain_keyword`;
- action-only route rules such as `sniff`, `hijack-dns`, and `reject`;
- logical route rules;
- DNS rules with `query_type`, `match_response`, `client_subnet`, and action fields;
- subscription extensions such as `outbounds[].filter`;
- placeholder node pools such as `{all}`;
- tags containing spaces, punctuation, Chinese text, and emoji;
- large `route.rule_set` arrays;
- old-version DNS formats;
- old-version inbound/listen fields.

## Higgsfield Reference Node Pixel Requirements

The visual target is the Higgsfield Canvas selected-node screenshot supplied by the user, not a generic React Flow card. The current first-pass implementation is insufficient if it renders ordinary rectangular config cards without the external title, external typed ports, internal bottom controls, and top-right Inspector behavior below.

Reference screenshot coordinate model, using the supplied crop as approximately `2048 x 1572`:

- Canvas background: nearly black surface with a subtle dotted grid. Dots are low-contrast, evenly spaced, and remain visible behind the selected node and Inspector. When the right Inspector is open, a vertical dashed boundary line appears near the Inspector side of the canvas.
- Selected node rectangle: large dark rounded rectangle, roughly square in the crop (`x ~= 98`, `y ~= 445`, `w ~= 984`, `h ~= 984`). The node body has a muted `1px` border. Selection adds a bright blue outline around the full rectangle and small square blue resize handles at all four corners.
- Node header above the rectangle: the node type title is outside the rectangle, aligned to the top-left of the node, with a small type icon followed by muted label text, for example a video icon plus `Video Generation`. For SBC this becomes an icon plus labels such as `Inbound / Tun`, `Outbound / Selector`, `DNS / Server`, or `Route / Rule`.
- Left side ports: typed upstream handles are outside the left edge, not inside the rectangle. They form a vertical stack of circular dark buttons overlapping the left margin. The reference stack shows image, video, audio waveform, and text ports with distinct icon colors. For SBC, each node type must declare its accepted upstream port types, and only compatible handles appear.
- Right side ports: downstream handles are outside the right edge. The reference output is a circular lime video port centered on the right side with a curved lime edge leaving the node. For SBC, outbound/reference-producing handles must visually encode the target type, such as traffic, route action, DNS server, selector member, endpoint, certificate provider, or service.
- Main node body: the center is intentionally sparse and dark. The placeholder/title text sits in the upper-left interior, large and muted. SBC nodes should use this area for the highest-signal summary only, not dense form fields.
- Bottom-left internal action: a square rounded `+` button sits inside the node body above the bottom toolbar. It is not an external port. For SBC it opens the compatible-node add menu or adds the next common object from the selected node's declared downstream registry.
- Bottom internal toolbar: controls sit inside the node along the bottom edge, not in the Inspector. The reference shows a model pill with icon and text, a compact stepper pill, a settings/sliders icon button, and a bright lime primary action button on the bottom-right. For SBC:
  - left pill shows the concrete sing-box subtype or primary tag, for example `tun-in`, `selector`, `urltest`, or `dns-remote`;
  - middle compact controls show key count/status such as candidate count, rule count, or validation warnings;
  - settings icon opens/focuses the Inspector;
  - primary action is context-specific, such as validate, add rule, add outbound member, or open JSON focus, but must use the same visual weight.
- Hover/selection affordances: add/delete/connect controls are visible on hover or selection only. Moving over a node should reveal compatible upstream/downstream add affordances; moving over an added child/edge should reveal delete/disconnect affordances.
- Edge style: connected edges are curved, color-coded by port type, and attach to the circular side handles. Edges should not imply that ordered rule tables are sourced from canvas position.

Pixel-level node decomposition from the updated user screenshot:

- The selected node is not a compact card. It is a large canvas object with a separate title strip and an inner work surface. On desktop, the selected SBC node should target a large readable footprint (`~310-360px` wide in current app scale, with stable aspect ratio and `min-height >= 260px`) rather than the old `230px x 124px` config card.
- Above the rectangle: the title row floats outside the node body, offset by roughly one icon height. It contains only the type icon and type label. It must not be inside the selected blue outline.
- Around the rectangle: the blue selected outline hugs the node body only. Four small square blue handles sit exactly on the outline corners. If these handles are missing, the node is not considered visually complete.
- Left of the rectangle: upstream ports sit on the outside edge as circular buttons, vertically stacked, centered on the node's left boundary and partially overlapping the node margin. These are not toolbar buttons and not text chips. Each port is icon-first and type-colored. The icon must represent the connectable upstream node type, not the current node. For example, a `Route` node's left port uses the `Inbound` icon because traffic enters from inbound nodes.
- Regression rule: a node implementation fails this requirement if it simply reuses the selected node's own icon for every left-side port. Group membership ports belong on member outbound nodes as explicit `Selector` / `URLTest` upstream types; selector/urltest group nodes themselves should not show a duplicate self-icon just to represent their own member list.
- Right of the rectangle: downstream ports sit outside the right edge, normally one primary circular lime handle for the active outgoing relation plus additional compatible handles when the node type exposes them. Curved edges must leave from these circles, not from the rectangle center. The icon must represent the connectable downstream node type, such as `Route Rule`, `Outbound`, `DNS Server`, or selector member, not the current node.
- Inside the upper body: the content is sparse. The first large text block is a muted summary/placeholder, placed near the top-left interior with generous empty space around it. Dense schema editing belongs in the Inspector or ordered tables.
- Inside the lower-left body: a square rounded `+` button floats above the toolbar. It creates a compatible adjacent entity through domain commands, not by directly adding arbitrary React Flow nodes.
- Inside the bottom edge: the toolbar is a horizontal strip of compact pills. From left to right it should read as: subtype/tag pill, count/status pill, settings/sliders icon, and a high-emphasis lime action at the bottom-right. The toolbar is inside the rectangle and must remain visible when selected.
- On hover/selection: compatible add affordances and delete/disconnect controls appear, but the default non-hover state remains clean. Controls added by hover must not shift the node body size.

Pixel-level Inspector decomposition from the updated user screenshot:

- The Inspector is a floating top-right overlay on the canvas. It is not the third column of the page grid and not below the canvas. It sits over the dotted canvas, with its own rounded border and elevated shadow.
- The Inspector top aligns close to the canvas top padding. Its right edge aligns close to the viewport/canvas right padding. Desktop target width is approximately `360-420px` in SBC scale, with mobile falling back to an overlay sheet.
- Header row: icon and selected node type on the left, close `X` on the right. This header mirrors the canvas node header.
- Directly below the header: a large rounded primary editor/summary block. It contains muted text and an internal `+` affordance when the node type supports adding/importing primary data.
- Field rows: label left, compact dark pill/input right. The visual rhythm must match the reference panel rows, not full-width stacked form fields unless mobile width forces it.
- Footer/action area: a bright lime full-width primary action appears near the bottom of the Inspector content. It is visually stronger than ordinary field controls.

Definition of done for this visual slice:

- A selected representative node must show the external title row, blue outline, four corner handles, left external port stack, right external port stack, internal bottom-left add button, internal bottom toolbar, and hover-only management controls.
- Opening settings or selecting a node must show the floating top-right Inspector with header, primary editor block, field rows, and footer action.
- Playwright must assert the DOM hooks for these pieces and capture at least one screenshot artifact for review.
- The implementation is incomplete if it only renders generic cards, a fixed right grid column, dense form fields inside nodes, or canvas nodes whose React Flow data becomes the config source of truth.

The expanded Inspector target is the right-side floating panel in the screenshot:

- Position: top-right overlay panel on the canvas, not a page section and not a card nested inside another card. In the crop it starts near `x ~= 1468`, `y ~= 10`, with `w ~= 580`, `h ~= 918`, rounded outer corners, and a slightly lighter dark background than the canvas.
- Header row: icon plus selected node type title on the left and a close `X` on the right. The header mirrors the selected canvas node type.
- Top editor block: a large rounded dark textarea/dropzone sits immediately under the header, with muted placeholder text and a small `+` button inside the block. For SBC this becomes the node's primary summary/edit block, such as server address/import text for proxy nodes or rule expression summary for route/DNS rules.
- Field rows: labels are left-aligned and controls are right-aligned. Controls are compact dark pills with icons or values. The reference rows are `Model`, `Duration (s)`, `Resolution`, `Aspect Ratio`, `Generate Audio`, and `Batch Size`; SBC rows must map to node schema fields and keep the same two-column rhythm.
- Footer action: a full-width bright lime primary button spans the Inspector bottom content area. For SBC this should be the selected node's main action or validation/apply command. It must not replace canonical config updates, which still go through domain commands.
- Inspector scope: editing a field updates canonical `SingBoxConfig` / domain state first, then the node view re-renders from derived state. The Inspector must not mutate React Flow node data as source of truth.

Design QA gates for this goal:

- Playwright must capture the selected-node and open-Inspector state for at least one representative fixture.
- DOM/CSS assertions must verify external node header, left/right external port containers, internal bottom toolbar, selected outline, corner handles, and top-right Inspector presence.
- The visual implementation review must explicitly compare the SBC node against this screenshot and record deviations in this document or the goal completion notes.

## Performance Requirements

Apply `vercel-react-best-practices` during implementation and review.

Required performance gates:

- graph derivation uses memoized derived data and Map/Set indexes where repeated lookups happen;
- large fixture import does not trigger repeated full validation loops per node;
- hover/drag state stays out of broad canonical config subscriptions;
- JSON editor remains lightweight or deferred;
- E2E large fixture import completes under a documented local and CI threshold;
- production bundle size change is reported.

## Acceptance Criteria

- `fixtures/external/manifest.json` contains at least 200 accepted unique fixtures.
- Every accepted fixture passes its manifest gates.
- Quarantined/rejected fixtures are listed separately with reasons and do not count toward 200.
- Official version-matched binary checks run for every fixture that claims official compatibility.
- Data-driven unit/integration tests cover all accepted fixtures.
- Playwright E2E covers representative external fixtures and has no console/runtime errors.
- Existing `pnpm release:check` still passes.
- New external hardening check passes locally and in CI.
- Documentation records source repos, fixture counts, failure categories, and fixed bug classes.
- Final commits are signed, pushed, GitHub Verified, and CI green.

## Done Definition

This goal is complete only when:

- at least 200 accepted external fixtures pass;
- no accepted fixture is display-broken;
- no accepted fixture loses unknown fields during round-trip;
- no accepted fixture causes graph/render/Inspector/table/JSON crash;
- CI runs both normal release checks and external fixture checks;
- the result is documented in a compatibility report;
- signed commits are pushed and verified.

If fewer than 200 accepted passing fixtures are available after discovery, the goal is blocked, not complete.
