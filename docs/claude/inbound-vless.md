| Field | Type | Required | Notes |
|---|---|---|---|
| `type` | `"vless"` | yes | fixed |

Total protocol-specific top-level fields: **1 required array** (`users`), **3 shared object sections** (`tls`, `multiplex`, `transport`), **listen block** (~14 fields).

### users[] item schema

| Field | Type | Required | Notes |
|---|---|---|---|
| `name` | string | — | display name |
| `uuid` | string (UUID) | **yes** | auth credential |
| `flow` | string | — | sub-protocol; only valid value is `"xtls-rprx-vision"` |

`flow` is meaningful only when TLS Reality is enabled. When `flow` is `""` or absent, standard VLESS is used without XTLS.

### tls (inbound shape) — with Reality sub-object

Key inbound-only fields:

| Field | Notes |
|---|---|
| `enabled` | boolean toggle |
| `server_name` | SNI |
| `alpn` | string[] |
| `min_version` / `max_version` | `1.0`–`1.3` |
| `cipher_suites` | string[] |
| `curve_preferences` | string[] (since 1.13.0) |
| `certificate` / `certificate_path` | PEM chain |
| `key` / `key_path` | server private key (server-only) |
| `client_authentication` | `no`/`request`/`require-any`/`verify-if-given`/`require-and-verify` (since 1.13.0) |
| `client_certificate` / `client_certificate_path` / `client_certificate_public_key_sha256` | mTLS client validation (server-only, since 1.13.0) |
| `kernel_tx` / `kernel_rx` | kTLS (Linux 5.1+, TLS 1.3 only, since 1.13.0) |
| `acme` | sub-object (domain, email, provider, challenges, dns01_challenge) |
| `ech` | sub-object with `enabled`, `key`, `key_path` |
| `reality` | sub-object — **server-only** — see Reality fields below |

#### reality sub-object (inbound/server-only)

| Field | Type | Required | Notes |
|---|---|---|---|
| `enabled` | boolean | yes | must be `true` to activate Reality |
| `handshake` | object | **yes (server-only)** | destination server + Dial Fields; e.g. `{ "server": "google.com", "server_port": 443 }` |
| `private_key` | string | **yes (server-only)** | generated by `sing-box generate reality-keypair` |
| `short_id` | string[] | **yes** | hex string(s), 0–8 digits |
| `max_time_difference` | duration string | — | max client/server clock skew; disabled if empty |

The Reality sub-object is one of the most important VLESS-specific TLS configurations. It requires special handling: the `handshake` sub-object itself embeds Dial Fields (detour, bind_interface, etc.). The outbound side uses a different Reality shape (`public_key` + `short_id`, no `handshake`, no `private_key`).

Outbound-only TLS fields NOT applicable here: `disable_sni`, `insecure`, `utls`, `fragment*`, `record_fragment`, `certificate_public_key_sha256`, outbound-role `client_certificate`/`client_key`.

### multiplex (inbound shape)

| Field | Notes |
|---|---|
| `enabled` | boolean |
| `padding` | boolean; if true, non-padded connections rejected |
| `brutal` | sub-object (TCP Brutal) |

Outbound-only multiplex fields NOT applicable: `protocol`, `max_connections`, `min_streams`, `max_streams`.

### transport (v2ray-transport)

Type discriminated by `type` field. Available types: `http`, `ws`, `quic`, `grpc`, `httpupgrade`.

| type | Unique fields |
|---|---|
| `http` | `host[]`, `path`, `method`, `headers{}`, `idle_timeout`, `ping_timeout` |
| `ws` | `path`, `headers{}`, `max_early_data`, `early_data_header_name` |
| `quic` | (no extra fields) |
| `grpc` | `service_name`, `idle_timeout`, `ping_timeout`, `permit_without_stream` |
| `httpupgrade` | `host`, `path`, `headers{}` |

Note: `flow: "xtls-rprx-vision"` and `transport` are mutually exclusive in practice — XTLS Vision bypasses the transport layer. The UI should surface a warning if both are set simultaneously.

### Inspector.tsx — users[] rendering

The `users` field is listed in `inboundHandledFields` (line 186 of Inspector.tsx) as a handled field. However:

## Priority Findings

### P0 — users[] array is invisible in Inspector

`users` is in `inboundHandledFields` (suppresses AdvancedScalarFields fallback) but no Inspector rendering block for `ref.kind === "inbound"` with `entityType === "vless"` exists. A user who opens the VLESS inbound Inspector **cannot see or edit the users list at all**. The field is created correctly at node creation time (commands.ts default), but is permanently invisible after that.

**Required fix:** Add a VLESS inbound users block in Inspector.tsx for `ref.kind === "inbound" && entityType === "vless"`. At minimum, a `JsonField` textarea. Better: a structured repeater with `name` (text), `uuid` (text with UUID generation button), and `flow` (select: `""` / `"xtls-rprx-vision"`).

### P0 — Reality sub-object has no UI whatsoever

The `reality` object inside `tls` is the primary differentiator for VLESS (especially for censorship circumvention). The current TLS panel in Inspector.tsx (lines 894–905) exposes only 8 scalar fields and has **zero support** for the `reality` sub-object. Users cannot:
- Enable Reality (`reality.enabled`)
- Configure the handshake destination (`reality.handshake.server` / `reality.handshake.server_port`)
- Enter the private key (`reality.private_key`)
- Set short IDs (`reality.short_id`)
- Set max time difference (`reality.max_time_difference`)

Without a structured Reality panel, VLESS's most important real-world configuration path is inaccessible through the UI. Users must fall back to raw JSON editing.

**Required fix:** Add a `reality` section to the TLS panel (or as a separate conditional sub-panel), gated on `ref.kind === "inbound" && type === "vless"` (and potentially `trojan`). Fields: `reality.enabled` (checkbox), `reality.handshake.server` (text), `reality.handshake.server_port` (number), `reality.private_key` (text with "Generate" action), `reality.short_id` (text / list), `reality.max_time_difference` (text).

### P1 — `insecure` shown in inbound TLS panel

The shared TLS panel renders `insecure` (line 898). `insecure` is documented as **client only** — it is not a valid inbound TLS field. Showing it in the inbound Inspector is misleading and may cause users to add it to an inbound configuration where it has no effect.

**Required fix:** Gate `insecure` on `ref.kind !== "inbound"` in the TLS field spec, or split the TLS field list into separate inbound/outbound definitions.

### P1 — `key_path` missing from TLS panel

The inbound TLS panel does not expose `key_path` (nor `key`). For inbound vless using standard TLS (non-ACME), users must provide the server private key — this is a required complement to `certificate_path`. Without `key_path` in the structured panel, users must fall to AdvancedScalarFields (if it gets through) or raw JSON.

**Required fix:** Add `key_path` (and optionally `key`) to the shared TLS panel field list, at minimum for `ref.kind === "inbound"`.

### P1 — Outbound-only multiplex fields shown for inbound

The multiplex panel renders `protocol`, `max_connections`, `min_streams`, `max_streams` (lines 919–924). These are **outbound-only** fields per the official multiplex inbound spec (inbound only supports `enabled`, `padding`, `brutal`). Showing them in the inbound inspector is misleading and may produce invalid config if a user sets them.

**Required fix:** Split multiplex field specs into inbound and outbound variants, or conditionally suppress outbound-only fields when `ref.kind === "inbound"`.

### P1 — `flow` / `xtls-rprx-vision` has no UI

The `flow` field on each `users[]` item is the XTLS Vision toggle — a prominent VLESS feature. Without it exposed in the users repeater, users cannot enable XTLS Vision through the UI, and must manually edit JSON. This is directly paired with P0 (users invisibility), but worth calling out explicitly.

### P1 — listenSharedFields incomplete

`listenSharedFields` in Inspector.tsx is missing: `tcp_multi_path`, `disable_tcp_keep_alive`, `tcp_keep_alive`, `tcp_keep_alive_interval`, `udp_fragment`, `detour`. These unhandled listen fields fall through to `AdvancedScalarFields` for raw scalar rendering. This affects all inbounds including VLESS, not VLESS-specific, but VLESS inherits the gap.
